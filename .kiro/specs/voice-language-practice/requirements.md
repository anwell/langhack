# Requirements Document

## Introduction

Voice Language Practice is a mobile-first language speaking practice application that enables users to have real-time voice conversations with an AI in a target language. The app provides structured role-play scenarios, live transcription, and a hands-free experience suitable for use while driving. The system uses a multi-agent architecture: a Conversation_Agent (Gemini Live streaming API) for real-time voice role-play, a Teacher_Agent for post-session evaluation, and a Scenario_Agent for intelligent scenario generation using Apify as a scraping tool. The Scenario_Agent can scrape TripAdvisor via Apify to generate travel-related practice scenarios grounded in real destination data (attractions, restaurants, hotels, activities). The frontend is built with React Native and Expo, and the backend uses FastAPI deployed as a stateless serverless function on AWS Lambda behind API Gateway. Session transcripts are stored locally on the device and uploaded to Box.com for cloud backup.

## Glossary

- **App**: The React Native mobile application built with Expo that serves as the user interface
- **Backend**: The FastAPI server that manages authentication, scenario data, and ephemeral token generation
- **Gemini_Live_API**: Google's Gemini Live streaming API used for real-time voice conversation via WebSocket
- **Ephemeral_Token**: A short-lived authentication token generated by the Backend for establishing a WebSocket connection between the App and Gemini_Live_API
- **Scenario**: A predefined conversational context for role-play practice (e.g., ordering at a restaurant, asking for directions)
- **Target_Language**: The language the user is practicing speaking
- **Source_Language**: The user's native or fluent language, defaulting to English. The Source_Language is used for UI feedback text, translations, vocabulary explanations, and Teacher_Agent responses
- **Transcript**: A text record of the spoken conversation between the user and the AI
- **Session**: A single voice conversation instance between the user and the AI within a selected Scenario
- **WebSocket**: A persistent bidirectional communication channel between the App and Gemini_Live_API for streaming audio
- **AWS_Lambda**: The AWS Lambda function that hosts the Backend as a serverless, stateless compute unit
- **API_Gateway**: The AWS API Gateway that exposes the Backend endpoints as HTTPS routes and routes requests to AWS_Lambda
- **Box**: The Box.com cloud storage service used to persist Session Transcripts as files
- **Box_File**: A single Transcript file stored in Box, containing the full conversation text, Session date, and Scenario title
- **Apify**: The Apify web scraping platform used as a tool by the Scenario_Agent to scrape phrasebook and travel guide content from the web
- **Apify_Actor**: A configured Apify actor invoked by the Scenario_Agent to scrape external content from specified URLs
- **Scenario_Agent**: A separate AI agent responsible for generating new practice Scenarios. The Scenario_Agent uses Apify_Actors as a tool to scrape raw content, then processes and transforms that content into well-structured, pedagogically useful Scenarios appropriate for the user's Target_Language and proficiency context
- **TripAdvisor_Scraper**: An Apify_Actor specifically configured to scrape TripAdvisor destination data including attractions, restaurants, hotels, and activities for a given city or region
- **Travel_Scenario**: A Scenario generated from real TripAdvisor destination data that incorporates actual place names, addresses, and contextual vocabulary to create realistic travel-related language practice situations
- **Destination**: A city or region where the Target_Language is spoken, used as the geographic focus for TripAdvisor scraping and Travel_Scenario generation
- **Conversation_Agent**: The Gemini Live voice AI that role-plays the Scenario partner during an active Session, handling real-time spoken interaction with the user
- **Teacher_Agent**: A separate AI agent invoked after a Session ends that evaluates the user's speaking performance based on the Transcript and produces structured feedback
- **Session_Feedback**: The structured evaluation produced by the Teacher_Agent, containing a Session_Score, Session_Pass_Fail result, performance highlights, areas for improvement, specific corrections, suggested vocabulary or phrases, suggested next Scenarios, and a Lesson_Plan
- **Suggested_Scenarios**: A list of recommended Scenarios included in Session_Feedback that the Teacher_Agent suggests the user practice next, based on weaknesses or gaps identified during the Session
- **Lesson_Plan**: A lightweight, actionable study guide included in Session_Feedback that the Teacher_Agent generates based on the user's weaknesses identified during the Session, containing specific focus areas and practice phrases
- **Post_Session_Review_Screen**: The screen displayed after a Session ends where the user can view the Session_Feedback generated by the Teacher_Agent
- **Session_Report**: A comprehensive document uploaded to Box containing the full Transcript plus the Teacher_Agent analysis (vocabulary highlights, grammar points, corrections, and Lesson_Plan)
- **Session_Score**: A numerical evaluation (0-100) assigned by the Teacher_Agent representing the learner's overall performance in achieving the Scenario's intended communication outcome
- **Session_Pass_Fail**: A binary evaluation (pass or fail) assigned by the Teacher_Agent indicating whether the learner achieved the Scenario's intended communication outcome
- **Score_Threshold**: The minimum Session_Score value (60) at or above which the Session_Pass_Fail is set to pass
- **Achievement_Badge**: A visual icon or badge displayed on the Post_Session_Review_Screen to celebrate milestones such as first session completed, consecutive session streaks, or perfect scores

## Requirements

### Requirement 1: Scenario Selection

**User Story:** As a language learner, I want to choose from a list of role-play scenarios, so that I can practice speaking in realistic conversational contexts.

#### Acceptance Criteria

1. THE App SHALL display a list of available Scenarios with a title and a description of no more than 150 characters for each
2. WHEN the user selects a Scenario, THE App SHALL navigate to the Session screen with the selected Scenario context loaded
3. THE Backend SHALL provide an API endpoint that returns the list of available Scenarios, where each Scenario includes at minimum a unique identifier, a title, and a description
4. THE App SHALL ship with a preloaded list of Scenarios bundled in the application package as a fallback
5. WHEN the App starts, THE App SHALL request the latest Scenario list from the Backend and update the locally cached list upon a successful response
6. IF the App fails to load the list of Scenarios from the Backend, THEN THE App SHALL display the locally cached or preloaded Scenario list without showing an error to the user
7. IF the locally cached Scenario list is empty and the Backend is unreachable, THEN THE App SHALL display a message indicating no Scenarios are currently available and offer a retry option

### Requirement 2: Voice Session Initialization

**User Story:** As a language learner, I want to start a voice conversation session quickly, so that I can begin practicing without complex setup.

#### Acceptance Criteria

1. WHEN the user starts a Session, THE Backend SHALL generate an Ephemeral_Token for the Gemini_Live_API with an expiry time of no more than 60 seconds
2. WHEN the App receives an Ephemeral_Token, THE App SHALL close any existing WebSocket connection and establish a new WebSocket connection to the Gemini_Live_API within 10 seconds
3. WHEN the WebSocket connection is established, THE App SHALL send the Scenario context, Target_Language, and Source_Language as the system prompt to the Gemini_Live_API
4. IF the WebSocket connection fails or the connection is not established within 10 seconds, THEN THE App SHALL display an error message indicating the connection failure and offer a retry option up to a maximum of 3 consecutive attempts
5. IF the Backend is unreachable or the Ephemeral_Token generation fails, THEN THE App SHALL display an error message indicating the service is unavailable and offer a retry option
6. WHILE the Session is being initialized, THE App SHALL display a loading indicator until the WebSocket connection is established or an error occurs

### Requirement 3: Real-Time Voice Conversation

**User Story:** As a language learner, I want to have a spoken conversation with the AI in my target language, so that I can practice pronunciation and conversational flow.

#### Acceptance Criteria

1. WHILE a Session is active, THE App SHALL stream the user's microphone audio to the Gemini_Live_API via WebSocket
2. IF microphone permission is not granted, THEN THE App SHALL display an error message indicating microphone access is required and SHALL NOT start the Session
3. WHILE a Session is active, THE App SHALL play the AI's audio responses received from the Gemini_Live_API through the device audio output
4. WHEN the user taps the stop button, THE App SHALL end the Session and close the WebSocket connection within 3 seconds
5. WHILE a Session is active, THE App SHALL display a visual indicator showing the conversation is in progress
6. IF the WebSocket connection is lost during an active Session, THEN THE App SHALL notify the user that the connection was lost and offer options to reconnect or end the Session
7. WHILE the AI audio response is playing, THE App SHALL allow the user to speak, interrupting AI playback and sending the new user audio to the Gemini_Live_API

### Requirement 4: Conversation Transcription

**User Story:** As a language learner, I want the conversation to be transcribed in real time, so that I can review words and phrases after the session.

#### Acceptance Criteria

1. WHILE a Session is active, THE App SHALL display each Transcript entry within 2 seconds of the corresponding speech being detected
2. THE App SHALL label each Transcript entry with the speaker (user or AI)
3. WHEN a Session ends, THE App SHALL persist the Transcript locally on the device along with the Session date and associated Scenario title
4. THE App SHALL provide a screen where the user can view past Session Transcripts listed in reverse chronological order, each identified by date and Scenario title
5. IF transcription text is unavailable during an active Session, THEN THE App SHALL display an indicator that transcription is temporarily unavailable while continuing the voice Session

### Requirement 5: Hands-Free Operation

**User Story:** As a language learner, I want to use the app hands-free, so that I can practice while driving or doing other activities.

#### Acceptance Criteria

1. WHEN a Session is started, THE App SHALL begin streaming audio and playing AI responses without requiring additional taps
2. WHILE a Session is active, THE App SHALL use voice activity detection to automatically determine when the user has finished speaking and send the audio to the Gemini_Live_API, requiring no user interaction for turn-taking
3. WHILE a Session is active, THE App SHALL keep the screen awake and continue audio streaming if the App is moved to the background
4. IF an audio interruption occurs during an active Session (e.g., incoming phone call), THEN THE App SHALL pause the Session and resume streaming when the interruption ends without requiring user interaction
5. WHILE a Session is active, THE App SHALL route audio output through the currently active audio device (device speaker or connected Bluetooth audio)

### Requirement 6: Language Configuration

**User Story:** As a language learner, I want to select which language I am practicing and which language I receive feedback in, so that the AI responds in the correct language and I can understand explanations in my native language.

#### Acceptance Criteria

1. THE App SHALL provide a language selection option displaying at least 2 Target_Languages before starting a Session
2. WHEN the user selects a Target_Language, THE App SHALL persist the selection locally so that it remains selected across app restarts and future Sessions
3. WHEN a Session is started, THE Backend SHALL include the selected Target_Language in the system prompt sent to the Gemini_Live_API
4. IF the user attempts to start a Session without a Target_Language selected, THEN THE App SHALL prompt the user to select a Target_Language before proceeding
5. THE App SHALL provide a source language selection option defaulting to English
6. WHEN the user selects a Source_Language, THE App SHALL persist the selection locally so that it remains selected across app restarts and future Sessions
7. WHEN a Session is started or feedback is requested, THE App SHALL send the Source_Language to the Backend alongside the Target_Language for use in system prompts, feedback, and translations

### Requirement 7: Serverless AWS Deployment

**User Story:** As a developer, I want the Backend deployed on AWS Lambda behind API Gateway, so that the system remains serverless, stateless, and scales automatically without managing infrastructure.

#### Acceptance Criteria

1. THE Backend SHALL be deployed as a single AWS_Lambda function that handles all API requests
2. THE API_Gateway SHALL expose all Backend endpoints as HTTPS routes and forward requests to the AWS_Lambda function
3. THE Backend SHALL operate statelessly, storing no session data, user data, or application state on the server between requests
4. THE Backend SHALL not use DynamoDB or any server-side database for persistence
5. WHEN the AWS_Lambda function receives a request, THE Backend SHALL complete processing and return a response within 29 seconds to remain within API_Gateway timeout limits
6. THE Backend SHALL read all configuration values (API keys, Box credentials, Apify tokens) from environment variables configured on the AWS_Lambda function
7. IF the AWS_Lambda function encounters a cold start, THEN THE Backend SHALL complete initialization and respond to the request without returning an error to the client

### Requirement 8: Box.com Session Report Storage

**User Story:** As a language learner, I want a comprehensive session report (transcript plus teacher analysis) saved to Box.com automatically after each session, so that I can access a complete record of my practice and feedback from any device.

#### Acceptance Criteria

1. WHEN the Teacher_Agent completes evaluation of a Session, THE App SHALL send the complete Transcript and the Session_Feedback to the Backend for upload to Box as a single Session_Report
2. WHEN the Backend receives a Session_Report upload request, THE Backend SHALL create a Box_File in the user's designated Box folder containing the full Transcript text, Session date, Scenario title, Session_Score, Session_Pass_Fail result, performance highlights, areas for improvement, corrections, suggested vocabulary, and Lesson_Plan
3. WHEN the Box_File is created successfully, THE Backend SHALL return the Box file URL to the App
4. THE App SHALL store the Box file URL alongside the locally persisted Transcript so that the user can access the Box_File later
5. THE App SHALL display a "View in Box" link on each Transcript entry in the transcript history screen that has an associated Box file URL
6. WHEN the user taps the "View in Box" link, THE App SHALL open the Box file URL in the device's default browser or the Box mobile app if installed
7. IF the Box upload fails, THEN THE Backend SHALL return an error response to the App and THE App SHALL retain the Transcript and Session_Feedback locally and display a message indicating the cloud backup failed
8. IF the Box upload fails, THEN THE App SHALL allow the user to retry the upload from the transcript history screen
9. THE App SHALL persist Transcripts and Session_Feedback locally on the device regardless of whether the Box upload succeeds, ensuring no data loss
10. THE Box upload SHALL happen automatically as part of the post-session flow after the Teacher_Agent evaluation completes, without requiring any user action

### Requirement 9: Scenario Agent Generation

**User Story:** As a language learner, I want new practice scenarios intelligently generated from real phrasebook and travel guide content, so that I always have fresh, pedagogically useful, and contextually relevant conversation topics to practice.

#### Acceptance Criteria

1. THE Backend SHALL provide an API endpoint that invokes the Scenario_Agent to generate new Scenarios for a given Target_Language
2. WHEN the Backend invokes the Scenario_Agent, THE Backend SHALL pass the Target_Language, Source_Language, and any available proficiency context as input parameters
3. THE Scenario_Agent SHALL use Apify_Actors as a tool to scrape phrasebook and travel guide content from the web, deciding which sources to scrape based on the Target_Language and proficiency context
4. WHEN the Apify_Actor returns raw scraped content, THE Scenario_Agent SHALL process and transform the content into well-structured Scenario objects containing a unique identifier, title, description, and key vocabulary or phrases relevant to the Scenario
5. THE Scenario_Agent SHALL ensure generated Scenarios are pedagogically appropriate by filtering irrelevant content, structuring conversational context, and producing descriptions that guide the learner toward useful practice
6. THE Scenario_Agent SHALL produce Scenarios in the same JSON format as the static Scenario list endpoint
7. THE Backend SHALL return the Scenario_Agent output directly in the API response without persisting generated Scenarios in a server-side database
8. THE Backend SHALL operate statelessly when invoking the Scenario_Agent, maintaining no server-side state between generation requests
9. THE App SHALL be able to fetch Scenarios from the Scenario_Agent endpoint as an alternative or supplement to the static Scenario list
10. WHEN the App receives Scenario_Agent-generated Scenarios, THE App SHALL merge the new Scenarios with the locally cached list, avoiding duplicate entries based on the Scenario unique identifier
11. IF the Scenario_Agent fails, the Apify_Actor returns no results, or the Backend is unreachable, THEN THE Backend SHALL return an empty Scenario list and a status indicating the generation was unsuccessful
12. IF the Scenario_Agent invocation fails, THEN THE App SHALL continue to display the locally cached or preloaded Scenario list without showing an error to the user

### Requirement 10: Teacher Agent Post-Session Evaluation

**User Story:** As a language learner, I want an AI teacher to automatically evaluate my speaking performance after each session with a score and pass/fail result, so that I can immediately understand how well I did and track my improvement over time.

#### Acceptance Criteria

1. WHEN a Session ends, THE App SHALL automatically invoke the Teacher_Agent evaluation without requiring any user action
2. WHEN the automatic evaluation begins, THE App SHALL send the complete Transcript, the Target_Language, the Source_Language, and the list of locally cached Scenario identifiers and titles to the Backend for evaluation by the Teacher_Agent
3. WHEN the Backend receives a feedback request, THE Backend SHALL invoke the Teacher_Agent with the Transcript text, Target_Language, Source_Language, and available Scenario list as input
4. THE Teacher_Agent SHALL analyze the Transcript and produce Session_Feedback containing eight sections: Session_Score, Session_Pass_Fail, performance highlights, areas for improvement, specific corrections with the original phrase and a corrected alternative, suggested vocabulary or phrases the user could have used, Suggested_Scenarios, and a Lesson_Plan
5. THE Teacher_Agent SHALL evaluate the user's grammar, vocabulary usage, pronunciation patterns inferred from the Transcript text, and conversational flow
6. THE Teacher_Agent SHALL provide all feedback text, corrections, explanations, and vocabulary translations in the Source_Language
7. THE Teacher_Agent SHALL produce Suggested_Scenarios containing between 1 and 3 recommended Scenarios for the user to practice next, based on weaknesses or gaps identified in the Transcript
8. WHEN a Suggested_Scenario matches an existing Scenario from the available Scenario list, THE Teacher_Agent SHALL reference the existing Scenario by its identifier and title
9. WHEN no existing Scenario adequately addresses an identified weakness, THE Teacher_Agent SHALL suggest a new Scenario idea with a proposed title and description
10. THE Teacher_Agent SHALL produce a Lesson_Plan containing between 1 and 5 actionable study items, where each item includes a focus area description and a list of up to 5 specific phrases or grammar patterns to practice
11. THE Lesson_Plan SHALL be a lightweight study guide limited to actionable next steps derived from the current Session, not a full curriculum
12. WHEN the Teacher_Agent completes evaluation, THE Backend SHALL return the Session_Feedback including Session_Score, Session_Pass_Fail, Suggested_Scenarios, and Lesson_Plan to the App in a structured JSON format
13. WHEN the App receives Session_Feedback, THE App SHALL display the feedback on the Post_Session_Review_Screen organized by the eight sections defined in criterion 4
14. WHEN the App receives Session_Feedback, THE App SHALL persist the Session_Feedback including Session_Score, Session_Pass_Fail, Suggested_Scenarios, and Lesson_Plan locally on the device alongside the associated Transcript
15. THE App SHALL allow the user to view past Session_Feedback from the transcript history screen for any Session that has associated feedback
16. WHEN the user taps a Suggested_Scenario that references an existing Scenario, THE App SHALL navigate to the Session screen with that Scenario context loaded
17. THE Backend SHALL operate statelessly when invoking the Teacher_Agent, maintaining no persistent user progress tracking or historical performance data on the server
18. IF the Teacher_Agent invocation fails or the Backend is unreachable, THEN THE App SHALL display a message indicating feedback is temporarily unavailable and offer a retry option
19. IF the Teacher_Agent invocation fails, THEN THE App SHALL retain the Transcript locally and allow the user to request feedback again later from the transcript history screen
20. THE Teacher_Agent SHALL assign a Session_Score between 0 and 100 based on the learner's overall performance in achieving the Scenario's intended communication outcome, considering grammar accuracy, vocabulary range, conversational flow, and task completion
21. THE Teacher_Agent SHALL assign a Session_Pass_Fail result of pass when the Session_Score is at or above the Score_Threshold of 60, and fail when the Session_Score is below 60
22. WHEN the Session ends and the automatic evaluation begins, THE App SHALL display a loading state on the Post_Session_Review_Screen indicating that the Teacher_Agent is analyzing the Session

### Requirement 11: TripAdvisor Scraping for Travel Scenarios

**User Story:** As a language learner, I want practice scenarios based on real travel destinations where my target language is spoken, so that I can prepare for actual trips with realistic place names, restaurants, and attractions.

#### Acceptance Criteria

1. WHEN the Scenario_Agent receives a generation request that includes a Destination parameter, THE Scenario_Agent SHALL invoke the TripAdvisor_Scraper Apify_Actor to retrieve destination data for that Destination
2. WHEN the Scenario_Agent receives a generation request without a Destination parameter but with a Target_Language associated with a known geographic region, THE Scenario_Agent SHALL select a default Destination where the Target_Language is commonly spoken
3. THE TripAdvisor_Scraper SHALL retrieve data including attraction names, restaurant names, hotel names, activity descriptions, and addresses for the specified Destination
4. WHEN the TripAdvisor_Scraper returns destination data, THE Scenario_Agent SHALL transform the scraped content into Travel_Scenarios that incorporate real place names, addresses, and contextually relevant vocabulary
5. THE Scenario_Agent SHALL generate Travel_Scenarios covering at least three categories: asking for directions to real attractions, ordering food at real restaurants, and checking in at real hotels
6. THE Scenario_Agent SHALL include in each Travel_Scenario a key_vocabulary list derived from the scraped content containing place-specific terms, common phrases for the scenario type, and relevant proper nouns from the Destination
7. THE Scenario_Agent SHALL produce Travel_Scenarios in the same JSON format as the static Scenario list endpoint, with the source field set to "generated"
8. IF the TripAdvisor_Scraper returns no results, times out, or encounters an error, THEN THE Scenario_Agent SHALL fall back to generating generic travel Scenarios for the Target_Language without real destination data
9. THE Backend SHALL accept an optional destination parameter in the scenario generation request to allow the user to target a specific city or region
10. WHEN the App displays the scenario generation interface, THE App SHALL allow the user to optionally specify a Destination city for travel scenario generation
11. THE Scenario_Agent SHALL filter scraped TripAdvisor content to exclude irrelevant entries such as closed establishments, entries with insufficient data, or content not in the Target_Language region
12. THE Scenario_Agent SHALL limit TripAdvisor scraping to a maximum of 20 results per category to maintain reasonable response times and Apify usage costs

### Requirement 12: Gamified Post-Session Review UI

**User Story:** As a language learner, I want the post-session review to feel rewarding and game-like with animations, colors, and achievement badges, so that I stay motivated to practice regularly and celebrate my progress.

#### Acceptance Criteria

1. WHEN the Session_Feedback is received and the Session_Pass_Fail result is pass, THE Post_Session_Review_Screen SHALL display a celebratory animation (confetti or sparkles effect) around the pass result
2. WHEN the Session_Feedback is received and the Session_Pass_Fail result is fail, THE Post_Session_Review_Screen SHALL display an encouraging animation (gentle pulse or upward arrow effect) around the fail result with a motivational message
3. THE Post_Session_Review_Screen SHALL display the Session_Score prominently using a counting-up animation that increments from 0 to the final score value over a duration of 1 to 2 seconds
4. WHEN the Session_Score is 80 or above, THE Post_Session_Review_Screen SHALL display the score in green color
5. WHEN the Session_Score is between 60 and 79 inclusive, THE Post_Session_Review_Screen SHALL display the score in yellow or amber color
6. WHEN the Session_Score is below 60, THE Post_Session_Review_Screen SHALL display the score in red color
7. THE Post_Session_Review_Screen SHALL display the corrections section with a red background tint to visually distinguish errors
8. THE Post_Session_Review_Screen SHALL display the performance highlights section with a green background tint to visually distinguish positive feedback
9. THE Post_Session_Review_Screen SHALL display the suggested vocabulary section with a blue background tint to visually distinguish learning material
10. WHEN the learner completes a milestone, THE Post_Session_Review_Screen SHALL display an Achievement_Badge with an icon and label for the milestone achieved
11. THE App SHALL recognize the following milestones for Achievement_Badge display: first session completed, three consecutive sessions (streak), five consecutive sessions (streak), a Session_Score of 100 (perfect score), and ten total sessions completed
12. THE Achievement_Badge SHALL appear with a scale-up entrance animation when first displayed on the Post_Session_Review_Screen
13. THE Post_Session_Review_Screen SHALL display the Session_Pass_Fail result as a large, visually prominent badge or banner at the top of the feedback section before the score
14. WHILE the Teacher_Agent evaluation is in progress, THE Post_Session_Review_Screen SHALL display a progress animation indicating analysis is underway
